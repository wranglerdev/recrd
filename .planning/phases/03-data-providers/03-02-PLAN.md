Replace the stub `CsvDataProvider.cs` with full implementation per D-01, D-02, D-06, D-07.

**Constructor:** `CsvDataProvider(string filePath, string delimiter = ",")` storing both fields. Validate non-null.

**StreamAsync implementation using CsvHelper 33.1.0:**

1. Open file with `new StreamReader(_filePath, detectEncodingFromByteOrderMarks: true)` for BOM tolerance (DATA-01).

2. Create `CsvConfiguration(CultureInfo.InvariantCulture)` with:
   - `Delimiter = _delimiter` (DATA-01 configurable delimiter)
   - `BadDataFound = args => throw new DataParseException(args.Context.Parser.Row, args.Context.Parser.RawRecord ?? string.Empty, _filePath, $"Bad CSV data at line {args.Context.Parser.Row}: {args.Field}")` per D-07
   - `MissingFieldFound = null` — disable default behavior; we do manual column-count checking below

3. Create `using var csv = new CsvReader(reader, config)`.

4. Read header: `await csv.ReadAsync()` then `csv.ReadHeader()`. If `csv.HeaderRecord` is null, throw `DataParseException(1, "", _filePath, "CSV file has no header row")`.

5. Store `var headerCount = csv.HeaderRecord!.Length` and `var headers = csv.HeaderRecord!`.

6. Stream rows: `while (await csv.ReadAsync())` loop (NOT `GetRecordsAsync` — manual loop gives tighter control for column-count checking per research pitfall #2). Inside the loop:
   - Check cancellation: `cancellationToken.ThrowIfCancellationRequested()`
   - Wrap in try/catch for `CsvHelperException` and translate to `DataParseException` with `lineNumber: csv.Parser.Row`, `offendingLine: csv.Parser.RawRecord ?? string.Empty`, `filePath: _filePath`
   - Build `Dictionary<string, string>` by iterating `headers` array, calling `csv.GetField(i) ?? string.Empty` for each index (empty/null cells become `""`)
   - Validate column count: `csv.Parser.Count` vs `headerCount`. If mismatch, throw `DataParseException(csv.Parser.Row, csv.Parser.RawRecord ?? string.Empty, _filePath, $"Row {csv.Parser.Row} has {csv.Parser.Count} fields but header has {headerCount} columns")`
   - `yield return dict`

7. The `using` statements on `StreamReader` and `CsvReader` ensure proper disposal.

**Critical CsvHelper details:**
- Do NOT access `args.Context.Parser.Record` inside `BadDataFound` — causes stack overflow (research pitfall #6)
- Use `args.Context.Parser.RawRecord` only
- `CultureInfo.InvariantCulture` must be set on `CsvConfiguration` — no parameterless constructor exists in v33
- The `using` for CsvReader is necessary; do not use `await using` (CsvReader is `IDisposable`, not `IAsyncDisposable`)
